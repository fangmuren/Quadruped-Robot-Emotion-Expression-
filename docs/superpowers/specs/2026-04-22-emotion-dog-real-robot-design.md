# Emotion_dog 真机安全可运行版设计

## 目标

将 `Emotion_dog` 从“可以向机器人发动作命令的脚本集合”改造成“更接近真机可跑通的控制程序”。本次设计不修改 `scripts/auto_lcm_init.sh`，重点提升以下能力：

- 真机运行时的安全收尾，默认优先回到趴下状态
- 命令发送、heartbeat、反馈等待的运行正确性
- 情绪动作序列在成功、超时、异常、中断时的可预测行为
- 足够详细的运行日志，便于真机联调和排障
- 明确的测试与验收标准，判断程序是否真正比当前版本更接近“可跑通”

## 非目标

- 不重写情绪配置体系，不大幅改变 `config.py` / `emotion.py` 的数据结构
- 不改动 `scripts/auto_lcm_init.sh`
- 不引入复杂日志框架、外部监控系统或配置中心
- 不在本轮把所有情绪动作的具体数值调参到最优

## 当前结构

当前 `Emotion_dog` 的主执行链路为：

- `main.py`：CLI 入口与进程生命周期
- `motion_sequence.py`：情绪动作序列执行器
- `robot_control.py`：LCM 通信与 heartbeat
- `config.py` / `emotion.py`：情绪配置与运行策略
- `demo.py`：串行演示入口

当前版本已经具备基础命令发送能力，但仍偏向“能跑脚本”，距离“真机可调试、可安全收尾、可稳定定位问题”的版本还有差距。

## 设计概览

保留现有三层结构，但将 `robot_control.py` 升级为真机控制器，将 `motion_sequence.py` 升级为带结果判断的序列执行器，并让 `main.py` / `demo.py` 统一走安全退出流程。

### 分层职责

#### `robot_control.py`
负责：

- 管理 LCM TX/RX 与 heartbeat
- 保存最新反馈与反馈时间戳
- 提供单次动作、持续动作、安全收尾等高层控制接口
- 执行关键等待、超时判断、错误检测
- 输出控制层日志

不负责：

- 情绪编排策略
- CLI 参数处理
- demo 演示顺序策略

#### `motion_sequence.py`
负责：

- 将情绪配置翻译成执行步骤
- 按动作类型调用控制层安全接口
- 对失败、中断、stop 请求作出统一处理
- 输出序列层日志

不负责：

- LCM 细节
- 线程关闭细节
- 安全收尾动作的底层实现

#### `main.py` / `demo.py`
负责：

- 程序入口与参数解释
- 初始化控制器与序列执行器
- 捕获 `KeyboardInterrupt` 与顶层异常
- 调用统一停止与安全收尾流程
- 输出用户可见的总体状态日志

## 控制层设计

`robot_control.py` 从“薄封装”升级为“真机控制器”。

### 1. 连接与状态感知

控制器启动时输出：

- TX/RX URL
- LCM 通道名
- RX/TX 线程是否启动成功

运行时维护：

- 最新反馈对象
- 最近一次收到反馈的时间戳
- 最近一次发送的命令
- 当前是否处于安全停止流程中

这样可以区分“本地线程正常运行”和“机器人确实正在回反馈”。

### 2. 命令语义分离

控制层把动作分为两类：

#### 单次动作
例如：趴下、跳跃、短时情绪动作。

语义为：

- 发送动作命令
- heartbeat 维持该命令
- 等待反馈达到完成条件或超时
- 返回成功/失败结果

#### 持续动作
例如：loop 情绪、持续姿态、持续走动。

语义为：

- 进入目标动作
- 持续维持 heartbeat
- 运行期间监控 stop 请求与反馈活性
- 收到 stop 或超时后切安全收尾

这避免把所有动作都错误地简化成“发一下再 sleep”。

### 3. 反馈门控

关键动作执行时，控制层提供统一等待接口，条件可包括：

- `mode` 匹配目标
- `gait_id` 匹配目标
- `order_process_bar` 达到阈值
- 最近反馈未超时

如果等待超时，则动作直接返回失败，不允许序列继续往后执行。

### 4. 统一安全停止链路

新增统一语义，例如：

- `request_stop()`
- `safe_lie_down()`
- `shutdown_with_safe_lie_down()`

停止链路固定为：

1. 停止接收新的高层动作请求
2. 发送趴下动作
3. 在短时间内维持该命令 heartbeat
4. 等待趴下完成或达到安全超时
5. 关闭线程与 LCM 资源

目标是让“优先回到趴下”成为所有退出路径共享的默认行为。

### 5. 控制层日志

使用标准输出，统一日志格式，不引入复杂日志系统。

至少输出：

- 控制器启动/关闭
- 每次发命令的 `mode/gait_id/duration`
- 开始等待、等待成功、等待超时
- 收到 stop 请求
- 进入安全趴下、趴下完成或趴下超时
- 反馈解码异常、长时间无反馈、LCM 发布异常

## 序列层设计

`motion_sequence.py` 升级为带执行结果判断的真机序列执行器。

### 1. 每个 step 具备明确执行语义

尽量保持现有配置结构不变，但在执行层解释 step 类型：

- 单次动作：调用“发送并等待完成”的控制层接口
- 持续动作：调用“进入动作并保持”的控制层接口
- 停止动作：调用统一安全收尾

`duration` 不再等价于单纯 sleep，而是参与动作完成等待和超时计算。

### 2. 失败即中止整条序列

出现以下情况时，序列立即停止，不再进入后续步骤：

- 长时间无反馈
- 动作完成超时
- 控制层返回失败
- stop 请求发生在关键动作中

失败后统一进入安全趴下路径。

### 3. loop 情绪的可控退出

对于循环类型情绪：

- 每轮循环前后都检查 stop/失败状态
- 一旦收到 stop，不再进入下一步
- 退出循环后明确执行安全趴下

避免逻辑层退出后，机器人仍保持上一条持续动作。

### 4. 序列日志

序列层输出：

- 当前 emotion
- 当前 step 序号
- `mode/gait_id/duration`
- 当前 step 成功/失败/超时/被中断
- 如果触发安全收尾，打印是由哪一步触发

## 入口层设计

`main.py` 与 `demo.py` 统一走同一套生命周期管理。

### 1. 生命周期

统一流程：

1. 初始化控制器
2. 初始化序列执行器
3. 执行指定 emotion 或 demo
4. 在正常结束、异常、`Ctrl+C` 的任意路径下，进入统一停止流程

停止流程不直接 `close()`，而是：

1. 停止序列层继续发新动作
2. 调用控制层安全趴下
3. 安全收尾后关闭线程和 LCM

### 2. 用户可见行为

启动时打印：

- 当前 emotion
- 动作类型（single / loop）
- 是否进入循环模式

执行时打印：

- 当前 step
- 关键等待状态
- 是否进入 stop 或错误降级

结束时打印：

- 正常完成
- 被用户中断
- 因错误进入安全趴下
- 程序退出

### 3. Ctrl+C 行为固定

按下 `Ctrl+C` 后：

- 停止继续调度后续动作
- 触发统一安全趴下
- 完成后退出程序

用户应能稳定预期中断后机器人会优先趴下，而不是简单退出 Python 进程。

### 4. demo 行为

`demo.py` 也按真机标准处理：

- 每个情绪结束后确认已收尾
- 相邻情绪之间允许短暂稳定期
- 任意情绪失败时，立即停止整个 demo 并进入安全趴下

## 错误处理与超时策略

### 错误分级

#### 致命错误
直接进入安全趴下并结束当前执行：

- 长时间收不到反馈
- 关键动作超时
- LCM 发布/解码持续失败
- step 配置缺失关键字段导致无法执行

#### 非致命告警
打印日志，可继续执行：

- 单次反馈解码失败但后续恢复
- 可选字段缺失并使用默认值
- 单次等待偏慢但未超时

### 超时策略

按动作类型设置不同默认超时：

- recovery / stand：短超时
- 单次情绪动作：`duration + margin`
- loop 持续动作：重点监控反馈活性与 stop，而非等待完成
- lie down：给予更宽松超时，因为它是最终安全收尾

### 失败后的统一路径

一旦失败：

1. 记录失败原因
2. 停止后续 step
3. 进入安全趴下
4. 输出最终失败摘要

## 日志格式

采用可读、可 grep 的统一格式，例如：

- `[INFO] controller started tx=... rx=...`
- `[STEP] emotion=happy index=2/5 mode=62 gait=6 duration=3500`
- `[WAIT] mode=62 gait=6 timeout=4.0s`
- `[WARN] response decode failed`
- `[ERROR] action timeout, entering safe lie down`
- `[SAFE] lie down start`
- `[SAFE] lie down done`

日志必须满足三个要求：

- 能快速看出程序停在哪一步
- 能快速看出为什么失败
- 能快速看出是否进入安全趴下

## 配置校验

在执行 emotion 前进行轻量校验：

- `mode` 必填
- `gait_id`、`duration`、`velocity`、`rpy`、`position`、`step_height` 的结构正确
- 数值类型基本合理

这样能把部分错误提前到启动前，而不是在真机执行中暴露。

## 测试与验收标准

### 1. 静态层

- `main.py`、`demo.py`、`motion_sequence.py`、`robot_control.py` 可正常导入
- CLI 参数行为正常
- 配置校验能在启动前发现明显错误

### 2. 控制层自检

- 控制器启动后有明确日志
- 能报告 TX/RX 与线程状态
- 在没有机器人反馈时，不会假装成功，而是按超时路径失败

### 3. 真机动作验收

对每个 emotion，至少验证：

- 可以开始执行
- step 日志与配置一致
- 动作期间无明显卡死
- 正常结束后可进入安全趴下

对 loop 类型额外验证：

- `Ctrl+C` 后不会继续新一轮动作
- 会进入安全趴下
- 退出日志完整

### 4. 异常路径验收

至少人工验证：

- 执行中 `Ctrl+C`
- 无反馈超时
- step 配置错误

期望结果一致：

- 打印可定位日志
- 不继续执行后续动作
- 尝试安全趴下
- 最终退出，不残留后台执行线程

## 完成标准

本轮完成标准不是一次把所有情绪都调到最优，而是先达到以下状态：

- `happy` 或一个最常用情绪能够稳定跑通
- `Ctrl+C` 收尾稳定
- 无反馈超时路径稳定
- demo 中任意一个情绪失败时，整场演示能安全终止
- 日志足够定位“卡在哪一步、为什么失败、是否进入趴下”

## 推荐实施顺序

1. 先改 `robot_control.py`，建立统一安全接口和日志
2. 再改 `motion_sequence.py`，接入结果判断与失败中止
3. 然后改 `main.py` / `demo.py`，统一退出流程
4. 最后做基础静态验证和首个真机情绪验收

## 风险与取舍

- 为了尽量接近真机可跑通，本设计优先提升运行正确性与安全退出，而不是优先做更多功能
- 为了降低改动风险，本设计尽量兼容现有情绪配置，而不重写配置系统
- 趴下作为默认安全收尾动作更符合本轮用户要求，但在极端情况下仍可能因为机器人反馈异常导致收尾超时，因此安全收尾本身也必须带超时与日志
